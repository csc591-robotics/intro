# custom_map_builder

ROS 2 package for **loading an occupancy map (PGM + YAML)** in RViz and clicking **start / destination poses** for robot navigation.

- For **hand-crafted maps** (the `diamond_blocked`, `y_path`, etc. maps shipped with this package) everything works out of the box.
- For **maps generated by `world_to_map`** (warehouse, workshop_example, etc.) the runner script automatically reads the `.world_map.yaml` sidecar and aligns the map with Gazebo's coordinate frame.

This lives under `intro/src/custom_map_builder/` as part of the `intro` workspace.

---

## Build

From the `intro/` workspace:

```bash
cd /workspace/intro
source /opt/ros/humble/setup.bash
colcon build --packages-select custom_map_builder world_to_map
source install/setup.bash
```

**Maps folder:** Everything under `src/custom_map_builder/maps/` is copied into `install/.../share/custom_map_builder/maps/` on build. Add or change files there, then rebuild.

## Quick start — using the runner script

The runner script is the recommended way to launch. It handles everything automatically.

### For `world_to_map`-generated maps (warehouse, workshop_example, etc.)

```bash
cd /workspace/intro
source install/setup.bash

# RViz only (fastest for clicking points — no Gazebo needed):
LAUNCH_GAZEBO=false bash src/custom_map_builder/scripts/run_map_builder.sh warehouse

# Or with Gazebo open at the same time:
bash src/custom_map_builder/scripts/run_map_builder.sh warehouse
```

You can pass any map name from `src/world_to_map/maps/`:

```bash
LAUNCH_GAZEBO=false bash src/custom_map_builder/scripts/run_map_builder.sh workshop_example
LAUNCH_GAZEBO=false bash src/custom_map_builder/scripts/run_map_builder.sh test_zone
```

### For hand-crafted maps (diamond_blocked, y_path, etc.)

```bash
LAUNCH_GAZEBO=false bash src/custom_map_builder/scripts/run_map_builder.sh diamond_blocked
```

### What the runner does automatically

1. Resolves the map name to a `.yaml` file (searches `world_to_map/maps/` first, then `custom_map_builder/maps/`).
2. Looks for a sibling `.world_map.yaml` sidecar next to the YAML.
3. If found, reads `world_to_map_offset` from it and passes it as the `map → odom` TF so the map frame aligns 1:1 with Gazebo world coordinates.
4. Launches `map_server`, the static TF, RViz, and the click-echo node.

## Clicking start / destination points in RViz

1. Wait until the **Map** display shows your grid.
2. Select the **Publish Point** tool in the RViz toolbar (or press `P`).
3. Click anywhere on the map.
4. The terminal prints both coordinate systems:

```
frame="map"  MAP  x=9.42  y=13.10  |  GAZEBO WORLD  x=0.73  y=1.34
```

- **GAZEBO WORLD** coordinates → use these as source/destination for Nav2 and any navigation package. These are in the same frame Gazebo uses.
- **MAP** coordinates → only needed if you are computing pixel positions yourself (`pixel = map_coord / resolution`).

---

## How coordinate alignment works

The alignment is done through a single **`map → odom` static TF** (a translation, no rotation).

```
Gazebo world (0, 0)
    ↓   diff-drive plugin publishes odom → base_footprint
    odom frame  ←  identical to Gazebo world frame
    ↓   static TF: map → odom = (offset_x, offset_y)
    map frame   ←  where RViz and map_server live
```

For `world_to_map`-generated maps in `bottom-left` mode, the PGM was rasterized with all geometry shifted into the positive quadrant by `(offset_x, offset_y)`. That shift is stored as `world_to_map_offset` in the `.world_map.yaml` sidecar. Publishing `map → odom = (offset_x, offset_y)` re-glues the two frames so:

```
map_x  =  world_x  +  offset_x       (what you see in RViz)
world_x =  map_x   −  offset_x       (what Gazebo actually uses)
```

For hand-crafted maps the offset is `(0, 0)` — the map's `origin` in its YAML file already matches the Gazebo world corner, so no shift is needed.

You can read the offset programmatically from any package:

```python
import yaml, pathlib
meta = yaml.safe_load(pathlib.Path('warehouse.world_map.yaml').read_text())
dx, dy = meta['world_to_map_offset']   # e.g. [8.69, 11.76]

world_x = map_x - dx
world_y = map_y - dy
```

## Launch file arguments (advanced)

If you want to call the launch file directly instead of using the runner script:

```bash
ros2 launch custom_map_builder map_builder.launch.py \
  map_yaml:=/workspace/intro/src/world_to_map/maps/warehouse.yaml \
  map_offset_x:=8.690673 \
  map_offset_y:=11.755986 \
  launch_gazebo:=false \
  use_sim_time:=false
```

| Argument | Default | Meaning |
|---|---|---|
| `map_yaml` | built-in `default.yaml` | Absolute path to map YAML, or filename under `share/.../maps/` |
| `map_offset_x` | `0.0` | `map → odom` TF translation X — must equal `world_to_map_offset[0]` from sidecar |
| `map_offset_y` | `0.0` | `map → odom` TF translation Y — must equal `world_to_map_offset[1]` from sidecar |
| `launch_gazebo` | `true` | `false` = RViz + map only, no Gazebo |
| `launch_rviz` | `true` | `false` = headless (coordinates echoed to terminal only) |
| `use_sim_time` | `true` | Set `false` when Gazebo is not running |

## Generating custom maps

Generate a simple Y-shaped corridor PGM + YAML:

```bash
python3 src/custom_map_builder/scripts/generate_y_corridor_map.py
```

Then pass the generated YAML as `map_yaml`.

The shipped **default** map is a small empty room (`maps/default.pgm`) so the package runs without generating anything first.

## Troubleshooting

- **`TURTLEBOT3_MODEL` error:** Export before launching: `export TURTLEBOT3_MODEL=burger`. The runner script sets this automatically.
- **Map appears but robot is in the wrong place:** The `map_offset_x`/`map_offset_y` are wrong or zero. Check `world_to_map_offset` in the `.world_map.yaml` sidecar.
- **Clicks print `GAZEBO WORLD  x=0.00  y=0.00` for everything:** The offset parameters were not passed to the launch. Use the runner script instead of calling `ros2 launch` directly.
- **`install/setup.bash` not found:** Build first with `colcon build --packages-select custom_map_builder world_to_map`.
- **`use_rviz` unknown argument:** Your `turtlebot3_world.launch.py` may not support that argument. Edit `launch/map_builder.launch.py` and remove `'use_rviz': 'false'` from `launch_arguments`.
